# Phase 1 — User Model and Authentication

---

## 1.1 Design User model

Fields:

* `name`
* `email`
* `password`
* `role` (`admin`, `ngo`, `volunteer`)
* `skills` (optional for volunteer)
* `location`
* `availability`
* `createdAt`
* `updatedAt`

Optional later:

* uploaded files count
* assigned tasks
* profile image

---

## 1.2 Build User schema

Tasks:

* create Mongoose schema
* make email unique
* hash password before save
* add method to compare password

### Small pieces

* install bcrypt
* write pre-save hook
* write password compare method
* test user creation in DB

---

## 1.3 Register API

Route:

* `POST /api/auth/register`

Tasks:

* validate input
* check existing user
* hash password
* save user
* return token/cookie

### Validation

* name required
* email valid
* password min length
* role allowed values only

---

## 1.4 Login API

Route:

* `POST /api/auth/login`

Tasks:

* find user by email
* compare password
* generate auth token
* send response

---

## 1.5 Auth token/session setup

Choose one:

### Option A: JWT

Tasks:

* create token util
* sign token with user id
* send token in cookie or response
* verify token middleware

### Option B: Session auth

Tasks:

* install express-session
* configure session store
* attach user to session
* auth middleware from session

For hackathon, **JWT in httpOnly cookie** is simple and solid.

---

## 1.6 Protected routes middleware

Tasks:

* read token
* verify token
* load user
* attach user to `req.user`

Add:

* `isAuthenticated`
* `authorizeRoles(...roles)`

---

## 1.7 Current user API

Route:

* `GET /api/auth/me`

Tasks:

* return logged-in user profile
* exclude password

---

## 1.8 Logout API

Route:

* `POST /api/auth/logout`

Tasks:

* clear cookie or destroy session

---

## 1.9 Frontend auth integration

Tasks:

* create register form
* create login form
* connect to backend
* store auth state
* redirect after login
* show current user on dashboard
* add logout button

### Done when:

* user can register
* user can login

# Phase 2 — File Storage Design

---

## 2.1 Decide what “file storage” means

You need two layers:

### A. File metadata in MongoDB

Store:

* original filename
* saved filename
* path/url
* mime type
* size
* uploaded by
* processing status
* extracted text status
* embedding status

### B. Actual file binary storage

Initial options:

* local disk using `uploads/`
* Cloudinary / S3 later

For MVP:

* use **local storage**
* later abstract storage into service

---

## 2.2 Create File model

Fields:

* `originalName`
* `storedName`
* `filePath`
* `mimeType`
* `size`
* `uploadedBy`
* `status`

  * `uploaded`
  * `processing`
  * `processed`
  * `failed`
* `extractedText`
* `chunkCount`
* `embeddingStatus`
* timestamps

---

## 2.3 Create storage service abstraction

Make a service with methods:

* `saveFile()`
* `deleteFile()`
* `getFilePath()`

Even if local now, keep it abstract so cloud migration is easy.

# Phase 4 — Extract Text From Uploaded Files

This is the bridge between upload and RAG.

---

## 4.1 Decide supported formats

For MVP support:

* `.txt`
* `.pdf`

Later:

* `.docx`
* `.csv`

---

## 4.2 Build parser service

Create parser methods:

* `extractTextFromTxt(filePath)`
* `extractTextFromPdf(filePath)`

Libraries:

* for pdf: `pdf-parse`
* for txt: fs read

---

## 4.3 Processing pipeline

After upload:

1. detect file type
2. extract text
3. clean text
4. save extracted text in DB or separate collection
5. chunk text
6. embed chunks
7. store vectors

---

## 4.4 Text cleaning step

Tasks:

* remove extra whitespace
* normalize line breaks
* strip repeated headers if needed
* preserve meaningful structure

---

## 4.5 Update file statuses

Statuses:

* `uploaded`
* `extracting_text`
* `chunking`
* `embedding`
* `ready`
* `failed`

This is useful for frontend UI.

---

# Phase 5 — Chunking Strategy

---

## 5.1 Create chunking utility

Tasks:

* split long text into smaller chunks
* maintain overlap between chunks

Initial values:

* chunk size: 500–800 words or token-equivalent
* overlap: 100–150

---

## 5.2 Store chunks

Create `DocumentChunk` model:

* `fileId`
* `userId`
* `chunkIndex`
* `text`
* `embedding`
* maybe `metadata`

If vector DB stores vectors separately, still keep chunk text linked in DB.

---

## 5.3 Metadata per chunk

Useful fields:

* file name
* upload date
* user id
* section/page number if available
* category

This helps retrieval quality.

---

# Phase 6 — RAG on Uploaded Files

---

## 6.1 Choose embedding model

Options:

* OpenAI embeddings
* Gemini embeddings if available
* local embedding model later

For fastest MVP:

* use hosted embeddings API

---

## 6.2 Choose vector storage

Options:

* MongoDB Atlas Vector Search
* Pinecone
* Chroma
* Weaviate

If you already use MongoDB, **MongoDB Atlas Vector Search** is a clean choice.

---

## 6.3 Embedding pipeline

For each chunk:

1. generate embedding
2. store vector
3. link vector to chunk metadata

Tasks:

* create `embedText()` service
* batch process chunks
* handle rate limits
* retry failed embeddings

---

## 6.4 Retrieval service

Input:

* user question

Flow:

1. convert query to embedding
2. retrieve top-k similar chunks
3. pass chunks + question to LLM
4. generate grounded answer

Tasks:

* top-k search
* filter by user ownership
* optional file-specific filtering
* return citations/source chunk info

---

## 6.5 Answer generation service

Prompt should include:

* user question
* retrieved chunks
* instruction to answer only from context
* say insufficient information if context weak

Tasks:

* write prompt template
* build `askQuestionOverDocs(question, userId)` service
* parse LLM response

---

## 6.6 RAG API routes

Routes:

* `POST /api/rag/ask`
* `POST /api/rag/ask-file/:fileId`

Possible request body:

* `question`
* optional `fileIds`

---

## 6.7 Response format

Return:

* answer
* sources used
* file names
* relevant chunk previews

This makes the assistant more trustworthy.

---

## 6.8 Frontend RAG UI

Tasks:

* question input box
* answer panel
* source chips/cards
* loading state
* chat history optional

Views:

* ask across all my uploaded files
* ask only this file

# Phase 15 — Risks to Watch Early

## 1. Upload works but parsing fails

Fix by supporting only txt/pdf first.

## 2. RAG quality is poor

Fix with better chunking and metadata.

## 3. Answers hallucinate

Fix with strict prompt: answer only from retrieved context.

## 4. Large file delays

Fix using async processing and processing status.

## 5. Frontend and backend auth mismatch

Fix cookie/CORS setup early.
